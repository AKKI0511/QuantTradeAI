# Configuration Guide

This guide documents the configuration formats currently implemented in QuantTradeAI.

## Configuration Entry Points

QuantTradeAI supports two configuration entrypoints:

- `config/project.yaml` for the canonical `validate` and `research run` workflow
- Legacy YAML files under `config/` for `train`, `evaluate`, `backtest-model`, `live-trade`, and `validate-config`

`config/project.yaml` is generated by the CLI and is not checked into the repository by default.

## Canonical Project Workflow

Generate a starter config:

```bash
poetry run quanttradeai init --template research -o config/project.yaml
```

Supported templates are:

- `research`
- `llm-agent`
- `hybrid`

Validate the generated file:

```bash
poetry run quanttradeai validate -c config/project.yaml
```

Validation requires these top-level sections:

- `project`
- `profiles`
- `data`
- `features`
- `research`
- `agents`
- `deployment`

Validation writes artifacts under `reports/config_validation/<timestamp>/`:

- `resolved_project_config.yaml`
- `summary.json`

Run the canonical research workflow:

```bash
poetry run quanttradeai research run -c config/project.yaml
```

`research run` writes:

- Workflow summaries and runtime-generated configs under `runs/<timestamp>/`
- Training outputs under `models/experiments/<timestamp>/`

### Minimal project config shape

```yaml
project:
  name: research_lab
  profile: research

profiles:
  research:
    mode: research
  paper:
    mode: paper
  live:
    mode: live

data:
  symbols: ["AAPL", "MSFT"]
  start_date: "2018-01-01"
  end_date: "2024-12-31"
  timeframe: "1d"
  test_start: "2024-09-01"
  test_end: "2024-12-31"

features:
  definitions:
    - name: rsi_14
      type: technical
      params:
        period: 14

research:
  enabled: true
  labels:
    type: forward_return
    horizon: 5
    buy_threshold: 0.01
    sell_threshold: -0.01
  model:
    kind: classifier
    family: voting

agents: []

deployment:
  target: docker-compose
  mode: paper
```

## Legacy Config Validation

Validate all shipped runtime YAMLs:

```bash
poetry run quanttradeai validate-config --output-dir reports/config_validation
```

This validates the following files with the same schemas used by the CLI:

- `config/model_config.yaml`
- `config/features_config.yaml`
- `config/backtest_config.yaml`
- `config/impact_config.yaml`
- `config/risk_config.yaml`
- `config/streaming.yaml`
- `config/position_manager.yaml`

The command exits non-zero if any file fails validation.

## Legacy Model Config

`config/model_config.yaml` controls data loading, label generation, training splits, and trading defaults used by the legacy `train` and `evaluate` commands.

```yaml
data:
  symbols:
    - ticker: "AAPL"
      asset_class: equities
    - ticker: "ES"
      asset_class: futures
  start_date: "2015-01-01"
  end_date: "2024-12-31"
  timeframe: "1d"
  secondary_timeframes: ["1h", "30m"]
  cache_dir: "data/raw"
  cache_path: "data/raw"
  cache_expiration_days: 7
  use_cache: true
  refresh: false
  max_workers: 1
  test_start: "2024-09-01"
  test_end: "2024-12-31"

news:
  enabled: false
  provider: yfinance
  lookback_days: 30
  symbols: []
```

Implemented behavior to be aware of:

- `data.symbols` accepts either plain ticker strings or mappings with `ticker` and optional `asset_class`
- `asset_class` values are checked against `config/impact_config.yaml` when that file is present
- `data.test_start` and `data.test_end` must fall inside `data.start_date` and `data.end_date`
- If the requested test window validates but the downloaded data does not fully cover it, the training pipeline falls back to a chronological `training.test_size` split and writes a coverage report
- `news.provider` currently defaults to `yfinance`

Training-related fields still live in the same file:

```yaml
training:
  test_size: 0.2
  random_state: 42
  cv_folds: 5

trading:
  initial_capital: 100000
  position_size: 0.2
  stop_loss: 0.02
  take_profit: 0.04
  max_positions: 5
  transaction_cost: 0.001
```

Hyperparameter tuning uses `TimeSeriesSplit(n_splits=cv_folds)`.

## Features Config

`config/features_config.yaml` controls the legacy feature-engineering pipeline.

```yaml
pipeline:
  steps:
    - generate_technical_indicators
    - generate_multi_timeframe_features
    - generate_volume_features
    - generate_custom_features
    - generate_sentiment
    - handle_missing_values
    - remove_outliers
    - scale_features
    - select_features

price_features:
  - close_to_open
  - high_to_low
  - close_to_high
  - close_to_low
  - price_range

multi_timeframe_features:
  enabled: false
  operations:
    - type: ratio
      timeframe: "1h"
      base: close
      stat: last
```

Implemented validation rules include:

- `price_features` can be a list of enabled feature names or a mapping with booleans plus `sma_periods` and `ema_periods`
- `volume_features`, `volatility_features`, and `custom_features` accept the list-of-mappings style used in the shipped config
- `multi_timeframe_features.operations[].type` must be one of `ratio`, `delta`, `pct_change`, or `rolling_divergence`
- `rolling_divergence` operations require `rolling_window`
- Sentiment settings are optional and only affect the pipeline when `sentiment.enabled: true`

## Backtest and Execution Config

`config/backtest_config.yaml` controls the standalone CSV-based `backtest` command and provides execution defaults for `backtest-model`.

```yaml
data_path: data/backtest_sample.csv

execution:
  transaction_costs:
    enabled: false
    mode: bps
    value: 0
    apply_on: notional
  slippage:
    enabled: false
    mode: bps
    value: 0
    reference_price: close
  liquidity:
    enabled: false
    max_participation: 0.1
    volume_source: bar_volume
  impact:
    enabled: false
    model: linear
    alpha: 0.0
    beta: 0.0
    decay: 0.0
    spread: 0.0
    average_daily_volume: 0
```

Related config files:

- `config/impact_config.yaml` defines asset-class defaults such as `equities` and `futures`
- `config/risk_config.yaml` defines drawdown protection and turnover limits used by saved-model backtests and live trading

## Streaming and Live Trading Config

`live-trade` combines these files:

- `config/model_config.yaml`
- `config/streaming.yaml`
- `config/risk_config.yaml`
- `config/position_manager.yaml`

`config/streaming.yaml` contains the market-data providers plus health settings:

```yaml
streaming:
  symbols: ["AAPL"]
  providers:
    - name: "alpaca"
      websocket_url: "wss://stream.data.alpaca.markets/v2/iex"
      auth_method: "api_key"
      subscriptions: ["trades", "quotes"]
  health_check_interval: 30

streaming_health:
  metrics:
    enabled: true
    host: "0.0.0.0"
    port: 9000
  api:
    enabled: false
    host: "0.0.0.0"
    port: 8000
```

`config/position_manager.yaml` controls live position tracking:

```yaml
position_manager:
  risk_management:
    drawdown_protection:
      enabled: true
      max_drawdown_pct: 0.2
  impact:
    enabled: true
    model: linear
    alpha: 0.1
    beta: 0.05
  reconciliation:
    intraday: "1m"
    daily: "1d"
  mode: paper
```

`position_manager.mode` must be either `paper` or `live`.
